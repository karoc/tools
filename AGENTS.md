# Agent 规则

本文件约束所有在本仓库中工作的 Agent。执行任何代码变更前，必须先阅读并遵守 `PROJECT_CONSTRAINTS.md`。

## 核心原则

- 本仓库是各类小工具集合，不是单一应用仓库。
- 所有工具必须放在 `工具分类/工具名/` 结构下。
- 每个工具必须既能独立运行，也能被项目统一入口调用。
- 单个代码文件不得超过 700 行；超过或接近超过时必须拆分。
- 涉及接口调用时，必须按接口契约对齐请求字段、响应字段、错误字段和类型定义。
- 每次交付必须包含运行时验收，不能只做静态检查或代码阅读。
- 未覆盖事项按开源项目和广泛应用工程标准执行。

## 开始工作前

1. 查看仓库结构，确认是否已有统一入口、共享模块、测试框架和技术栈约定。
2. 查看目标工具所在目录；若是新增工具，先确认应放入哪个 `工具分类/工具名/` 路径。
3. 检查是否存在已有规则文件、README、脚本或 CI 配置，并优先沿用。
4. 识别变更是否涉及接口调用、统一入口、配置、运行命令或用户可见行为。

## 新增工具规则

新增工具必须满足以下交付项：

- 创建在 `工具分类/工具名/` 目录下。
- 提供独立运行入口。
- 接入项目统一入口。
- 提供 `README.md` 或等价说明，至少包含用途、安装、运行、测试、统一入口使用方式和运行时验收步骤。
- 若有示例输入输出，应放在该工具目录内的 `examples/` 或项目约定位置。
- 若有测试能力，应提供覆盖核心逻辑的测试。

目录示例：

```text
text/json-formatter/
  README.md
  src/
  tests/
  examples/
```

## 修改已有工具规则

- 保持该工具独立可用。
- 保持统一入口仍可调用该工具。
- 更新相关 README、示例、测试和验收步骤。
- 不把其他工具的逻辑塞入当前工具目录。
- 可复用逻辑应抽到共享模块，并保持共享模块职责单一。

## 文件行数规则

Agent 必须在提交前检查手写代码文件行数。

建议检查命令：

```bash
find . \
  -path './.git' -prune -o \
  -path './node_modules' -prune -o \
  -path './vendor' -prune -o \
  -path './dist' -prune -o \
  -path './build' -prune -o \
  -type f \
  \( -name '*.js' -o -name '*.jsx' -o -name '*.ts' -o -name '*.tsx' -o -name '*.vue' -o -name '*.py' -o -name '*.go' -o -name '*.rs' -o -name '*.java' -o -name '*.kt' -o -name '*.php' -o -name '*.rb' -o -name '*.sh' -o -name '*.css' -o -name '*.scss' -o -name '*.html' \) \
  -exec wc -l {} + | awk '$1 > 700 { print }'
```

若命令输出任何手写代码文件，必须拆分到 700 行以内再交付。

## 接口调用规则

涉及接口调用时，Agent 必须：

- 找到接口文档、OpenAPI schema、后端类型定义、示例响应或现有调用代码作为依据。
- 明确请求字段、响应字段、错误字段、分页字段、鉴权字段和时间格式。
- 使用类型、schema、DTO 或集中映射层表达字段契约。
- 补齐或更新接口相关测试。
- 在文档中记录必要的配置项和示例。

禁止：

- 凭字段名猜测实现；
- 在多个文件散落同一接口字段映射；
- 忽略错误响应；
- 用临时 mock 伪装真实接口已对齐。

## 统一入口规则

若仓库已有统一入口，新增或修改工具必须同步更新入口。

统一入口可以是以下形式之一，以项目已有实现为准：

- Web 首页、工具导航、路由表；
- CLI 命令集合；
- 桌面应用菜单；
- 服务端路由；
- 包导出入口；
- 其他项目约定入口。

入口接入必须保持名称、分类、参数、输出和错误提示清晰稳定。

## 验收规则

每次完成任务前，Agent 必须给出并尽量执行验收。

验收至少包含：

- 代码格式化或 lint；
- 类型检查或构建；
- 自动化测试；
- 运行时验收。

运行时验收必须实际运行相关工具或统一入口，并验证主要流程。

如果受环境限制无法执行运行时验收，最终回复必须明确说明：

- 哪个命令无法执行；
- 缺少什么环境或依赖；
- 已执行的替代验证；
- 用户需要如何继续完成验收。

## 文档规则

新增或变更工具时，必须同步更新相关文档：

- 工具用途；
- 独立使用方式；
- 统一入口使用方式；
- 配置项；
- 接口字段说明；
- 示例；
- 验收步骤。

文档应简洁、可执行、可复现。不要只描述概念而不给命令或操作路径。

## 代码质量规则

- 优先使用项目已有技术栈、目录风格和命名习惯。
- 变更范围应聚焦当前任务，避免无关重构。
- 共享代码必须有明确边界，不为单个调用点过早抽象。
- 错误处理要能帮助定位问题。
- 不提交密钥、令牌、私有凭据或本地机器路径。
- 引入依赖前应确认许可证、维护状态和必要性。
- 用户可见行为变更必须更新说明和测试。

## 最终交付说明

Agent 最终回复应包含：

- 修改了哪些文件；
- 是否满足 `工具分类/工具名/` 结构；
- 是否存在超过 700 行的手写代码文件；
- 已执行的静态、测试和运行时验收；
- 未执行项及原因。
