# 项目约束

本项目用于沉淀各类可独立运行的小工具。所有新增功能、重构、修复、文档与自动化脚本都必须遵守本文件约束。

## 1. 代码文件行数

- 单个代码文件不得超过 700 行。
- 若文件预计或已经超过 700 行，必须按职责拆分后再提交。
- 拆分优先级：
  - 业务流程与入口分离；
  - UI、状态、数据访问、接口调用、工具函数分离；
  - 平台适配层与核心逻辑分离；
  - 测试数据、示例数据、类型定义、常量配置独立成文件。
- 禁止通过压缩代码、删除必要换行、合并不相关职责来规避行数限制。
- 生成代码、构建产物、第三方依赖、锁文件不作为手写代码行数约束对象，但不应提交无必要的生成产物。

## 2. 目录结构

工具目录必须采用：

```text
工具分类/工具名/
```

示例：

```text
text/json-formatter/
image/png-compressor/
dev/jwt-debugger/
network/http-status-checker/
```

每个工具目录至少应清晰表达：

- 该工具所属分类；
- 该工具自身名称；
- 独立运行入口；
- 统一入口接入方式；
- 使用说明与验收方式。

禁止将多个不相关工具混放在同一工具目录内。共享代码应放在清晰命名的公共目录中，例如 `shared/`、`packages/` 或当前技术栈约定的共享模块目录。

## 3. 工具使用方式

每个工具必须同时满足：

- 可以独立使用：进入 `工具分类/工具名/` 后，可以按该工具文档直接安装、运行、测试或构建。
- 可以通过统一入口使用：项目级入口必须能够发现或调用该工具，且入口参数、路由、菜单或命令名称应稳定清晰。

新增工具时必须同步补齐：

- 独立运行说明；
- 统一入口接入说明；
- 必要的配置示例；
- 运行时验收步骤。

## 4. 接口调用约束

若工具存在任何接口调用，必须对齐接口字段：

- 请求字段必须与接口文档或实际服务契约一致。
- 响应字段必须按契约解析，不得猜测字段名。
- 字段映射应集中管理，避免在多个位置散落硬编码。
- 对必填字段、可选字段、默认值、枚举值、时间格式、数值单位和错误码进行明确处理。
- 接口变更时，必须同步更新调用代码、类型定义、测试用例和文档。
- 不允许静默吞掉接口错误；必须给出可定位的错误信息，并保留必要的上下文。

推荐在代码中使用类型定义、schema、DTO、OpenAPI 类型生成或运行时校验来保证字段对齐。

## 5. 验收标准

每项变更的验收标准至少包含运行时验收。

最低验收要求：

- 静态验收：格式化、类型检查、lint 或当前技术栈等价检查通过。
- 测试验收：有自动化测试时必须通过；新增关键逻辑时应补充测试。
- 运行时验收：必须实际启动或执行相关工具，验证主要流程可用。
- 文档验收：新增或变更工具时，使用说明、配置说明、入口说明和验收步骤保持同步。

运行时验收应记录：

- 执行命令；
- 运行环境；
- 核心操作路径；
- 期望结果；
- 实际结果；
- 无法运行时的明确原因与替代验证方式。

仅通过阅读代码或只运行构建命令，不满足本项目验收要求。

## 6. 开源与通用工程标准

未明确提到的事项，按开源项目和广泛应用的软件工程标准执行：

- 目录命名清晰、稳定、可迁移；
- README、示例、配置模板和变更说明随代码同步维护；
- 不提交密钥、令牌、私有凭据、个人本地路径或敏感数据；
- 依赖选择优先考虑维护活跃、许可证清晰、社区使用广泛的方案；
- 兼容性、可访问性、错误处理、日志、国际化和安全性按工具面向用户合理处理；
- 公共 API、CLI 参数、配置字段和输出格式保持向后兼容，破坏性变更必须说明；
- 代码风格遵循项目已有约定；没有既有约定时，采用当前语言生态的主流格式化与 lint 规则；
- 变更应尽量小而完整，避免夹带无关重构。

## 7. 推荐工具目录模板

```text
工具分类/工具名/
  README.md
  src/
  tests/
  examples/
```

不同技术栈可以调整内部目录，但不得破坏 `工具分类/工具名/` 这一级结构。
